ArcGIS Pro SDK Maps SDK for JavaScript Developer Workflow

Turning Shadows into Data:
From Visual Effect to Persistent GIS Feature Layer

Shadow analysis in ArcGIS has always been a visualization tool. This post shows how to flip that assumption — using ArcGIS Pro SDK for .NET to generate shadow geometry as actual spatial features, then consuming them as a queryable, time-enabled feature layer in the ArcGIS Maps SDK for JavaScript.

Introduction

Shadow analysis is one of the most visually compelling capabilities in ArcGIS. Urban planners use it to evaluate sunlight access, developers use it to simulate building impact, and environmental consultants use it to assess pedestrian comfort in dense neighborhoods.

But here is the problem that nobody talks about: every shadow you have ever generated in ArcGIS disappeared the moment you closed the scene.

The ShadowCast widget in the ArcGIS Maps SDK for JavaScript is a powerful tool. With its recent 5.0 release, it was even upgraded to a full analysis object. But its output is still a temporary visual overlay in a SceneView — it is not stored, not queryable, and not reusable across applications.

"What if a shadow could be stored in a geodatabase, queried with SQL, and consumed by any GIS application — just like a feature layer?"

That is the question this project explores. Instead of treating shadows as a rendering effect, we generate them as persistent geospatial polygons using the ArcGIS Pro SDK for .NET, and then load them as a standard FeatureLayer in the ArcGIS Maps SDK for JavaScript. The result is a shadow dataset that can be queried, filtered by time, shared across teams, and integrated into any workflow that consumes feature data.

💡
Why this is genuinely new The ArcGIS ecosystem has shadow visualization tools. It has never had a documented workflow for generating shadow geometry as persistent, queryable GIS features using the Pro SDK and consuming them cross-platform via the JS SDK. This is that workflow.

The Conceptual Shift

The core idea is simple but has surprisingly large implications. Shadow geometry is, fundamentally, a spatial operation: you project a polygon (a building footprint) in a direction determined by the sun angle, producing another polygon (the shadow footprint). That shadow polygon is geometry. And geometry is GIS data.

Instead of asking a rendering engine to draw the shadow, we ask the geometry engine to construct it — and then we save it.

Why This Has Never Been Documented

The Esri community has asked about programmatic shadow geometry before. A thread in the ArcGIS Pro SDK forums specifically asks: "Is there any way to obtain the shadow range?" — and the answer, as of that post, was effectively no. There is no documented API for extracting shadow geometry from ArcGIS Pro as a feature class.

This workflow fills that gap. Instead of waiting for a built-in API, we compute the shadow geometry ourselves using solar position mathematics and the ArcGIS Pro SDK's geometry engine, then write the results directly to a feature class.

⚠️
What existing tools do NOT do The ShadowCast widget (JS SDK) renders visual shadow overlays. ArcGIS Pro's Daylight tool animates sun position visually. Neither produces storable, queryable shadow polygons. This project introduces that capability as a developer workflow.

Implementation Walkthrough

The workflow runs in three phases: shadow geometry generation in ArcGIS Pro SDK, data persistence in a feature class, and web visualization using the ArcGIS Maps SDK for JavaScript.

Step 1 — Computing Sun Position (C# / ArcGIS Pro SDK)

For a given latitude, longitude, date, and time, we compute the sun's azimuth (horizontal angle from north) and altitude (angle above the horizon). These two values define the direction from which shadows are cast.

SunCalculator.cs — ArcGIS Pro SDK 
public class SunCalculator
{
    /// Returns solar azimuth (degrees from north) and altitude (degrees above horizon)
    public static (double azimuth, double altitude) GetSunPosition(
        double latitude,
        double longitude,
        DateTime utcDateTime)
    {
        // Day angle in radians
        double dayOfYear = utcDateTime.DayOfYear;
        double B = (2 * Math.PI / 365) * (dayOfYear - 1);

        // Solar declination (tilt of Earth's axis)
        double decl = 0.006918 - 0.399912 * Math.Cos(B)
                      + 0.070257 * Math.Sin(B);

        // Hour angle (how far the sun is from solar noon)
        double solarNoon = 12.0 - (longitude / 15.0);
        double hourAngle = (15.0 * (utcDateTime.Hour + utcDateTime.Minute / 60.0
                           - solarNoon)) * (Math.PI / 180);

        double lat = latitude * (Math.PI / 180);

        // Solar altitude above horizon
        double sinAlt = Math.Sin(decl) * Math.Sin(lat)
                      + Math.Cos(decl) * Math.Cos(lat) * Math.Cos(hourAngle);
        double altitude = Math.Asin(sinAlt) * (180 / Math.PI);

        // Solar azimuth (compass bearing of the sun)
        double cosAz = (Math.Sin(decl) - Math.Sin(lat) * sinAlt)
                     / (Math.Cos(lat) * Math.Cos(Math.Asin(sinAlt)));
        double azimuth = Math.Acos(Math.Clamp(cosAz, -1, 1)) * (180 / Math.PI);

        if (hourAngle > 0) azimuth = 360 - azimuth;

        return (azimuth, altitude);
    }
}

Step 2 — Generating Shadow Polygons (ArcGIS Pro SDK Geometry Engine)

Each building footprint polygon is projected in the direction opposite to the sun, scaled by the building height divided by the tangent of the solar altitude. This produces the shadow footprint on the ground plane.

ShadowGeneration.cs — ArcGIS Pro SDK 
public static Polygon GenerateShadowPolygon(
    Polygon buildingFootprint,
    double buildingHeightMeters,
    double solarAzimuthDeg,
    double solarAltitudeDeg)
{
    // Shadow length: height / tan(altitude) — longer when sun is low
    double altRad = solarAltitudeDeg * (Math.PI / 180);
    double shadowLength = buildingHeightMeters / Math.Tan(altRad);

    // Direction vector opposite to the sun
    double azRad = solarAzimuthDeg * (Math.PI / 180);
    double dx = -Math.Sin(azRad) * shadowLength;
    double dy = -Math.Cos(azRad) * shadowLength;

    // Offset the building footprint to create the shadow polygon
    GeometryEngine engine = GeometryEngine.Instance;
    Polygon offsetPolygon = (Polygon) engine.Move(buildingFootprint, dx, dy);

    // Union the footprint and its offset to fill the full shadow area
    Geometry shadow = engine.Union(buildingFootprint, offsetPolygon);
    return shadow as Polygon;
}

Step 3 — Writing to a Feature Class (ArcGIS Pro SDK)

Shadow polygons for each time step are written to a geodatabase feature class with a Timestamp field. This enables time-based filtering later in the web application.

FeatureWriter.cs — ArcGIS Pro SDK 
await QueuedTask.Run(() =>
{
    using RowBuffer rowBuffer = shadowFeatureClass.CreateRowBuffer();
    using EditOperation editOperation = new EditOperation();

    editOperation.Callback(context =>
    {
        foreach (var (shadowPolygon, timestamp) in shadowResults)
        {
            rowBuffer["SHAPE"]     = shadowPolygon;
            rowBuffer["Timestamp"] = timestamp;       // DateTime of this shadow
            rowBuffer["BuildingID"] = buildingId;      // Source building reference
            rowBuffer["ShadowLen"]  = shadowLength;     // Shadow length in meters

            shadowFeatureClass.CreateRow(rowBuffer);
        }
        context.Invalidate(shadowFeatureClass);
    }, shadowFeatureClass);

    await editOperation.ExecuteAsync();
    await Project.Current.SaveEditsAsync();
});

Step 4 — Loading as a FeatureLayer in the JavaScript SDK

Once the shadow dataset is published as a feature service (or exported as GeoJSON), it is loaded in the ArcGIS Maps SDK for JavaScript as a standard FeatureLayer. Because each shadow feature has a Timestamp attribute, the TimeSlider widget can filter them by hour — giving users a fully interactive temporal shadow explorer.

main.js — ArcGIS Maps SDK for JavaScript 
import SceneView    from "@arcgis/core/views/SceneView";
import WebScene     from "@arcgis/core/WebScene";
import FeatureLayer from "@arcgis/core/layers/FeatureLayer";
import TimeSlider   from "@arcgis/core/widgets/TimeSlider";

// 1. Load shadow data as a standard feature layer
const shadowLayer = new FeatureLayer({
  url: "https://services.arcgis.com/{org}/FeatureServer/0", // your published service
  timeInfo: {
    startField: "Timestamp"   // enables time-aware filtering
  },
  renderer: {
    type: "simple",
    symbol: {
      type: "polygon-3d",
      symbolLayers: [{
        type: "flat-color",
        color: [30, 30, 30, 0.55]   // semi-transparent dark shadow
      }]
    }
  },
  popupTemplate: {
    title: "Shadow at {Timestamp}",
    content: "Building: {BuildingID} | Shadow length: {ShadowLen}m"
  }
});

// 2. Add to a 3D scene
const view = new SceneView({
  container: "viewDiv",
  map: new WebScene({ portalItem: { id: "YOUR_CITY_SCENE_ID" } })
});
view.map.add(shadowLayer);

// 3. Add TimeSlider so users can navigate through hours
const timeSlider = new TimeSlider({
  container: "timeSliderDiv",
  view: view,
  timeVisible: true,
  loop: true,
  playRate: 1000  // advance one time step per second
});

view.whenLayerView(shadowLayer).then(lv => {
  // Example: query shadows at a specific hour
  const result = await lv.queryFeatures({
    where: "Timestamp >= '2024-06-21 08:00' AND Timestamp < '2024-06-21 09:00'",
    returnGeometry: true
  });
  console.log(`Shadows at 8 AM: ${result.features.length} features`);
});

Why This Matters for Urban GIS

Converting shadows from a visualization effect into GIS data unlocks capabilities that planners and analysts have never had access to through standard ArcGIS tools.

  1. 1

    Regulatory shadow compliance

    Cities like New York, London, and Amsterdam have shadow zoning laws. With shadow data stored as features, analysts can run spatial intersections to determine exactly how many hours per day a building casts a shadow onto protected spaces — parks, schoolyards, solar panels — and export that as a compliance report.

  2. 2

    Time-series shadow accumulation

    Because each hourly shadow is a stored feature, you can union shadows across a full day to compute cumulative shadow coverage — an operation you can do on a query result, not just a visual widget. This is useful for solar energy site assessment or heat island modeling.

  3. 3

    Cross-platform and enterprise reuse

    Shadow data stored in a geodatabase or feature service can be consumed by ArcGIS Pro analysis tools, ArcGIS Dashboards, ArcGIS StoryMaps, and any third-party application that reads GeoJSON or a feature service REST endpoint. The visualization is just one consumer of the data.

  4. 4

    Digital twin integration

    Shadow datasets can be integrated into digital twin environments such as ArcGIS Urban or CityEngine, where they serve as data inputs for energy modeling, microclimate simulation, and pedestrian comfort scoring — not just 3D visuals.

Limitations and Considerations

⚠️
Important caveats for production use This workflow produces ground-plane shadow projections from building footprints. It does not account for terrain elevation, existing shadow overlap between buildings, or atmospheric refraction. For regulatory-grade analysis, results should be validated against ArcGIS Pro's 3D solar tools.

Additional considerations:

  • Building height data must be accurate — errors in height produce proportionally incorrect shadow lengths.
  • Large cities with thousands of buildings will generate very large feature datasets for full-day analysis. Consider tiling or chunking by district.
  • Shadow overlap between adjacent buildings is not merged in this implementation — union operations add processing time for dense urban areas.
  • The solar position algorithm used here is a simplified analytical model. For sub-degree accuracy, integrate NOAA's Solar Calculator equations or SPA (Solar Position Algorithm) from NREL.

Future Directions

This is an initial workflow proof of concept. Several enhancements could make it production-ready:

  • GPU-accelerated geometry computation using parallel processing in the Pro SDK for large-city datasets.
  • Integration with ArcGIS Data Pipelines to schedule automated daily shadow dataset regeneration.
  • 4D streaming using time-aware vector tile layers for web performance at urban scale.
  • Integration with ArcGIS Urban to link shadow analysis directly to zoning rule evaluation.
  • Building-to-building shadow attribution — tracking which buildings are responsible for casting shadows onto which parcels.

Conclusion

Shadow analysis in ArcGIS has always answered the question: "What does this shadow look like?" This project asks a different question: "What can we do with shadow data?"

By shifting shadows from a rendering effect to a persistent, queryable feature layer, we unlock capabilities that standard ArcGIS shadow tools simply do not provide: SQL querying, time-series analysis, spatial joins, enterprise storage, and cross-platform reuse.

The ArcGIS Pro SDK for .NET provides the geometry engine to compute shadow polygons programmatically. The ArcGIS Maps SDK for JavaScript provides the visualization and query capabilities to consume them. Together, they form a pipeline that moves urban shadow analysis out of the visualization layer and into the data layer — where it can do real analytical work.
